Interoperability in Tech. Evolution—Thoughts on “Reticulate”

2019/03/11

Five days ago, the R package “reticulate”(1.11.1) was released on Cran, which allows us to directly call python objects in R1. Bloggers have been praising this ground-breaking development all the time for the enhanced interoperability between R and Python. Because with “reticulate,” we no longer need to jump between different IDEs in order to work on the same projects to leverage the complementarity between R and Python. This will certainly bring great change to the programming routine of many data scientists. As an occupational habit, I tend to think about the value creation and value capture in the dynamics of “industry” evolution. Here are some quick thoughts.

First of all, “reticulate” by Rstudio will benefit many data scientists who program in R and Python. R and Python are the two most popular programming languages in data science so far. The first decision for many newcomers in data science is always: “R vs. Python, which one should I choose?” Now this headache will be significantly lessened. Especially if we can overcome the “language barrier”, we won’t be stuck in the either-or. Second, on the community level, there have been quite a few “early adopters” who have already become bilinguals. Now they will leverage the new power of “interoperability” to boost their productivity using “reticulate.” Third, thinking from the technological (and strategic) perspective, we can obviously see the increased compatibility between two programming languages. What usually follows compatibility will be an increase in the adoption rate on the provider who opens up its door to embrace other players–in this case, Rstudio. Rstudio is the superpower in the R domain. Such a move will draw more python programmers to R and retain R programmers with the new empowerment of using Python on their platform. The “one-way” compatibility2 will greatly enhance the usability and desirability of the Rstudio as the platform provider.

The release of “reticulate” is a strategic move of Rstudio. But this is not the end of the story. The interoperability will simply enlarge the overlap of R and Python communities if we draw a Venn diagram. The Python-focused IEDs are also powerful. They may match the move of Rstudio. Much more likely, the reverse “recuticate” in Python will emerge from the large open source community, by innumerable, keen, and ingenious open source developers. Even if the one-way compatibility will continue, Python-focused IEDs will not be the loser. As R users can easily operate in Python in Rstudio, they are more likely to adopt the usage of Python and contribute to the Python world. This is similar to the case when Apple allow the windows to be installed on Mac and Kindle books on iPad. Both the competing platforms will benefit from the one-way compatibility, because the enlarged overlap in the Venn chart means increase two-way traffic flowing to both sides3. Let along the open source provides much more flexibility to the players involved. We cannot formulate the dynamics in a game-theoretic fashion, as the other players are “competing” quite differently. For example, Anaconda, the company who distribute many Python IDEs (including JupyterLab and Spyder), have also long been providing Rstudio on their cloud and GUI. I would see “reticulate” as a strategic, and also very harmonious move to create net benefits to the open source community at large.
